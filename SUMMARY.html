Architecture Summary: The Flaky Network

1. The Problem Space

The constraints heavily penalize delay and enforce a strict ≤ 2.0x bandwidth cap. Using standard ARQ (NACK/retransmissions) costs a full Round Trip Time (RTT), raising latency significantly. The architecture instead utilizes the allotted bandwidth cap entirely via Forward Error Correction (FEC) to achieve an exceptionally low playout delay score.

2. Data Framing Protocol (Wire Format)

To stay under the 2.0x cap, the Sender issues a unified 324-byte UDP Payload to the Relay.

Segment Size Description
seq 4 Bytes Big-endian unsigned int sequence number.
payload 160 Bytes The original raw audio frame data.
x_payload 160 Bytes XOR parity of payload[i-1] and payload[i-3].

Overhead Check: 324 / 164 = 1.975x. No feedback down-channel is used, safely clearing the 2.0x threshold constraint.

3. System-Level Decoding (Receiver)

The receiver's design allows out-of-order execution and aggressive cascading. When a packet arrives, it is logged into two structures (one for raw payloads, one for parities).

  • Single Drop Recovery: If frame i is dropped, frame i+1 arrives 20ms later containing (i XOR i-2). Since we have i-2, we immediately recover i.
  • Burst Drop Recovery: If both i and i+1 drop, frame i+3 provides enough algebraic context to deduce i+1, which recursively falls back to decode i.

A BFS (Breadth-First-Search) worklist loops through the data matrix, recovering packets and pushing them to the harness player natively on port 47020 long before the expiration deadline.
