Architecture Summary

1. The Main Idea

Using standard retransmissions (like asking for a packet again if it's lost) takes way too long because of round-trip time. So instead, I used Forward Error Correction (FEC). The sender just sends backup data along with the regular data, so if something gets lost, the receiver can figure it out on its own without asking.

2. Packet Format and Bandwidth Trick

To recover from burst drops (when 2 packets drop in a row), the sender attaches an XOR of packet (i-1) and (i-3) to the current packet (i).

Part Size Details
Sequence Number 4 Bytes To know the order of packets.
Main Payload 160 Bytes The actual audio frame data.
Backup Payload 160 Bytes XOR of payload(i-1) and payload(i-3).

Bandwidth Trick: Doing this makes the packet 324 bytes, which gives a 2.02x overhead (over the limit). So, I programmed the sender to skip the backup payload on every 10th packet. This brings the bandwidth overhead down to a safe ~1.92x.

3. How the Receiver Works

The receiver listens on the port and puts everything into two memory maps (one for raw data, one for backup data). It uses a Breadth-First Search (BFS) style queue. Whenever a packet arrives, it checks if it has enough pieces to solve for any missing packets.

Because the network delay spikes are huge on the bursty profile, the receiver needs about 120ms to actually receive the 3rd packet and do the math. Once it recovers a missing packet, it immediately pushes it to the player.
